# 文档分块策略

分块是 RAG 流程的第一步，它把长文档切成若干较小的片段，再分别向量化入库。分块的粒度直接决定检索质量的上限：切得太碎，一个完整的答案会被拆散到多个片段，召回时拿不全；切得太大，单个片段里混入大量无关内容，召回时噪声过多。因此分块策略的选择是 RAG 工程里最值得反复评测的环节之一。

## 递归段落分块

递归段落分块是一种边界感知的策略。它优先以空行分隔的段落为单位累积内容，当缓冲区接近目标大小时就切出一个片段。当单个段落超过目标大小时，递归段落分块会退化到按句子边界切分，以句号、问号、感叹号作为切点。这样切出的片段尽量不跨越自然语义边界，是大多数生产系统的默认选择。它的目标大小通常设在每片段五百一十二个 token 左右，对应大约一千个字符。

## 固定大小分块

固定大小分块是最朴素的做法，它按固定的字符窗口硬切，不看任何标点或段落。固定大小分块完全无视段落和句子边界，容易把一个完整的答案从中间切断。正因为如此，它常被当作评测里的负向对照：一个设计良好的评测台，应当能把固定大小分块的检索质量排到所有策略的最底部。如果它没有垫底，往往说明评测本身有问题，而不是这个策略意外地好。

## 重叠分块

重叠分块在固定大小的基础上，让相邻片段之间共享一段内容。重叠分块让相邻两个块共享一段内容，即使答案落在块的边界附近，也有更大概率被某一个块完整覆盖。重叠的代价是片段总数增加、存储和检索开销上升，因此重叠量需要在召回质量和成本之间权衡，常见的设置是在每个片段末尾保留三十二个 token 的重叠。把重叠分块和无重叠的固定大小分块放在一起对比，由于两者只差重叠这一个变量，截断率的差值就能干净地归因到重叠本身的贡献。

## 语义分块

语义分块更进一步，它先把文档切成句子，再用向量相似度判断相邻句子是否属于同一主题，在主题发生明显漂移的地方下刀。语义分块理论上能产出语义最连贯的片段，但它需要为每个句子额外计算向量，成本最高，且对相似度阈值很敏感。是否值得引入语义分块，应当用检索评测的实际数字来回答，而不是凭直觉假设它一定更好。
